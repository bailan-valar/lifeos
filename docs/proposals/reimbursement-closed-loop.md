# 报销功能流程闭环优化方案

> 生成日期：2026-06-12
> 模块：LifeOS 记账模块 - 退款/报销子系统
> 文档类型：产品方案

## 一、现状分析

### 1.1 当前退款能力

| 能力 | 状态 | 说明 |
|------|------|------|
| 创建退款账单 | ✅ 已实现 | 支持全额/部分退款，自动反转类型和账户方向 |
| 关联已有账单为退款 | ✅ 已实现 | 支持双向关联（支出→收入 / 收入→支出） |
| 退款列表标记 | ✅ 已实现 | 退款badge + 橙色左边框 + 实际金额显示 |
| 退款统计 | ⚠️ 部分 | 仅BudgetDashboard正确扣除退款，其他视图不处理 |
| 退款汇总视图 | ❌ 缺失 | 无独立的退款报表或汇总页面 |
| 退款状态追踪 | ❌ 缺失 | 无退款审批/处理流程 |
| 解除退款关联 | ❌ 缺失 | 一旦关联无法解除 |
| 累计退款校验 | ❌ 缺失 | 多次部分退款可能超过原账单金额 |
| 报销审批流 | ❌ 缺失 | 无多角色审批机制 |

### 1.2 已发现的 Bug

| 编号 | 问题 | 位置 |
|------|------|------|
| BUG-1 | 退款描述逻辑错误：模板字符串永远 truthy，`||` 后备值不会执行 | `useBills.ts:886` |
| BUG-2 | `useBillRefunds.startWatching()` 是空实现 | `useBillRefunds.ts:70-75` |
| BUG-3 | `BillRefundDialog` 硬编码样式值，未使用 Liquid Glass CSS 变量 | `BillRefundDialog.vue` 全局 |
| BUG-4 | `CategoryDetailView` 统计不排除退款，金额虚高 | `CategoryDetailView.vue` |

### 1.3 核心痛点

```
支出发生 → 手动记录退款 → 无追踪 → 无汇总 → 无闭环反馈
    │           │              │           │           │
    ▼           ▼              ▼           ▼           ▼
  支出账单   创建/关联退款   不知道哪些  不知道退款   不知道实际
  正常记录   完全靠记忆     还没退款    总额多少    花了多少
```

**用户故事：**
> "我出去团建花了 500 块，公司报销了 300 块。我想在 LifeOS 里记下来，但记录完退款后，没有任何地方能看到'哪些支出还没报销'、'这个月总共退了多少'。"

```typescript
// useBills.ts - createRefundBill 内增加
const existingRefunds = await getRefundsForBill(original.id)
const totalRefunded = sum(existingRefunds.map(r => r.amount))
if (add(totalRefunded, data.amount) > original.amount) {
```

```typescript
// useBills.ts - 新增方法
async function unlinkRefundBill(billId: string): Promise<void> {
  const db = await getDB()
  const doc = await db.bills.findOne(billId).exec()
  if (!doc) throw new Error('账单不存在')
  await doc.patch({
    isRefund: false,
    originalBillId: null,
    refundReason: null,
    updatedAt: now(),
  })
}
```

## 四、方案B：标准闭环（推荐进阶）

### 4.1 设计理念

**在方案A基础上，增加退款状态追踪字段，让用户能清楚看到每笔退款的进度。** 引入 `refundStatus` 字段，但保持退款仍是账单的附属属性，不创建独立集合。

### 4.2 数据模型变更

```typescript
// types/bill.ts - Bill 接口新增字段
interface Bill {
  // ... 现有字段

  // 退款状态追踪（新增）
  refundStatus?: 'none' | 'pending' | 'partial' | 'completed'
  // none: 无退款（默认）
  // pending: 有退款申请中
  // partial: 已部分退款
  // completed: 已全额退款

  // 退款期望（新增，可选）
  expectedRefund?: number        // 预期退款金额
  expectedRefundDate?: string    // 预期退款日期
  expectedRefundNote?: string    // 退款备注（如"公司报销"、"保险理赔"）
}

// 退款追踪表单
interface RefundTrackingFormData {
  billId: string
  expectedRefund: number
  expectedRefundDate: string
  expectedRefundNote: string
}
```

## 五、方案C：完整报销流

### 5.1 设计理念

**引入独立的报销单概念，支持多级审批、多笔关联、附件上传。** 适合小团队记账场景（如小公司的报销流程）。

### 5.2 数据模型变更

```typescript
// 新增集合：reimbursements
interface Reimbursement {
  id: string
  noteId: string

  // 基本信息
  title: string                // 报销单标题
  description: string          // 说明
  category: string             // 报销类型（差旅、餐饮、采购...）

  // 关联账单
  billIds: string[]            // 关联的支出账单ID列表
  totalAmount: number          // 报销总金额

  // 审批流
  status: 'draft' | 'submitted' | 'approved' | 'rejected' | 'paid' | 'cancelled'
  submittedAt: string
  approvedAt: string
  rejectedAt: string
  paidAt: string
  rejectReason: string

  // 审批人
  approverId: string           // 审批人账户ID
  approverName: string

  // 退款关联
  refundBillId: string         // 退款到账后关联的退款账单

  // 附件（URL列表）
  attachments: string[]

  createdAt: string
  updatedAt: string
}
```

## 六、方案D：智能报销

### 6.1 设计理念

**在方案B基础上增加智能匹配和自动化能力。** 当用户创建退款账单时，系统自动推荐可能关联的"待退款"支出。

### 6.2 核心特性

| 特性 | 说明 |
|------|------|
| 智能匹配 | 创建收入账单时，自动推荐金额相近的"待退款"支出 |
| 批量标记 | 支持批量将多笔支出标记为"待退款" |
| 逾期提醒 | 超过预期退款日期未收到退款时自动提醒 |
| 退款来源分析 | 按退款来源（公司、保险、商家）分类统计 |
| 导入关联 | CSV导入时自动识别退款性质的交易 |

### 6.3 智能匹配流程

```
用户创建收入账单 "公司报销 ¥500"
         │
         ▼
  系统自动扫描同笔记下的"待退款"支出
         │
         ├→ 找到匹配: "团建聚餐 ¥500" (金额一致)
         │   → 弹出建议: "是否将此收入关联为「团建聚餐」的退款？"
         │   → 用户确认 → 自动关联 + 更新状态
         │
         ├→ 找到近似: "出差打车 ¥480" (金额相近)
         │   → 列为候选，用户手动选择
         │
         └→ 无匹配
             → 正常保存，不弹出
```
